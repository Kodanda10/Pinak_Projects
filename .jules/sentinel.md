## 2025-05-13 - Mitigate Bandit B608 and Mass Assignment Risks in DatabaseManager
**Vulnerability:** The application was constructing dynamic SQL f-strings. While many were structurally safe (due to static mapping like `table_map.get(layer)`), the `DatabaseManager.update_memory` explicitly built dynamic `SET` clauses from user-controlled dictionary payloads (using `serialized.keys()`) without sanitizing whether keys were valid column names. This creates SQL Injection and Mass Assignment vectors.
**Learning:** Python f-strings in `sqlite3.Connection.execute()` triggers the `B608` Bandit rule. Even when input to an f-string is partially controlled by internal functions (e.g. static maps), unsanitized variable column names injected directly from API payloads (like `MemoryUpdate` models parsing to dict keys) introduce severe SQL vulnerabilities.
**Prevention:** Always validate that dynamic column names or table names constructed from user inputs are strictly valid string identifiers via `if not isinstance(key, str) or not key.isidentifier(): raise ValueError(...)`. For structurally safe queries dynamically formatted in code, affix inline `# nosec B608` comments directly to the string definition line to satisfy SAST tools without false positive noise.
