## 2024-05-14 - Fix Mass Assignment / SQL Injection in SQLite Memory Service
**Vulnerability:** The SQLite `DatabaseManager.update_memory` logic dynamically assembled an `UPDATE` query by blindly interpolating Python dictionary keys straight into the `SET` clause (`set_clause = ", ".join([f"{k} = ?" for k in serialized.keys()])`). Although the caller (`MemoryService.update_memory`) filtered out specific system keys using a denylist (`forbidden_keys`), the denylist failed to prevent attackers from injecting arbitrary strings like `content = ?, tenant = 'attacker' --` to hijack the update semantics and bypass multi-tenant protections.
**Learning:** Denylist approaches to input sanitization fall short when string boundaries aren't strictly verified for target serialization structures (e.g. SQL). Dynamic key binding requires deep strict validation when interpolating field names to prevent column pollution or statement rewriting.
**Prevention:** Database column references created dynamically from incoming JSON or dictionary payload keys must ALWAYS be validated with string token boundary checks (e.g., `isinstance(key, str) and key.isidentifier()`) before interpolation. Even better, abstract update logic should restrict dynamically updated fields against a definitive list of explicitly whitelisted model fields rather than a blacklist of known system IDs.
