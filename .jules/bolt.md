## 2023-10-25 - [Removed Synchronous VectorStore Saves in memory_service.py]
**Learning:** The Pinak Memory Service had a significant bottleneck where `self.vector_store.save()` was being called explicitly and synchronously inside the `add_memory`, `update_memory`, and `delete_memory` endpoints. Because the `VectorStore` class itself implements an asynchronous 5-second debounced save mechanism (`_schedule_save()`), explicitly calling `.save()` forced a full $O(N)$ index write to disk on every single insertion, resulting in $O(N^2)$ overall behavior during batch additions.
**Action:** Removed explicit `.save()` calls in the service layer, deferring entirely to the `VectorStore`'s internal debounced saving. This speeds up vector ingestion dramatically (from ~4.77s to ~0.51s for 1000 items in benchmarks). Future vector store implementations or backend modifications should always trust the store's internal debounce/persistence mechanisms rather than forcing synchronous disk writes on the main execution thread.
