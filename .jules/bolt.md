## 2023-11-20 - [SQLite Connection Pooling]
**Learning:** Initializing a new `sqlite3.connect` on every database interaction can cause severe lock contention and slowdowns in a highly multithreaded context, even with WAL mode and `busy_timeout` enabled. In our stress test, 10 threads doing 100 writes each took ~3.3s. By introducing a simple thread-safe connection pool using a list and threading lock, and reusing connections created with `check_same_thread=False` across the context manager (`DatabaseManager.get_cursor()`), the same 1000 concurrent writes dropped to ~0.1s.
**Action:** When working with SQLite in an environment experiencing high concurrency, always utilize a connection pool (either custom or via a library like SQLAlchemy) instead of recreating connections per transaction, and ensure PRAGMAs (`journal_mode=WAL`, `synchronous=NORMAL`) are applied only when connections are instantiated.
